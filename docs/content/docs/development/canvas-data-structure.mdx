# Canvas Data Structure

This document describes the current canvas frontend local storage data structure, image file storage, and cleanup mechanism.

## Current Storage Locations

Canvas projects are primarily stored in the browser locally:

- Canvas project JSON: `localForage`, database name `infinite-canvas`, storeName `app_state`, key `infinite-canvas:canvas_store`.
- My Assets JSON: `localForage`, database name `infinite-canvas`, storeName `app_state`, key `infinite-canvas:asset_store`.
- Image Blobs: Separate `localForage` instance, database name `infinite-canvas`, storeName `image_files`.
- Video and other media Blobs: Separate `localForage` instance, database name `infinite-canvas`, storeName `media_files`.

Canvas JSON does not persistently store large base64 images or videos. Image nodes, video nodes, assistant images, and asset media only store display URL, `storageKey`, and metadata; the actual Blob is read via `storageKey`.

## Canvas Project Structure

Each canvas project is a `CanvasProject`:

```ts
type CanvasProject = {
  id: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  nodes: CanvasNodeData[];
  connections: CanvasConnection[];
  chatSessions: CanvasAssistantSession[];
  activeChatId: string | null;
  backgroundMode: "lines" | "dots" | "blank";
  viewport: { x: number; y: number; k: number };
};
```

Field descriptions:

- `id`: Canvas project ID, generated by the frontend.
- `title`: Canvas name.
- `createdAt` / `updatedAt`: ISO strings.
- `nodes`: Canvas node list.
- `connections`: Node connection list.
- `chatSessions`: Right-side canvas assistant sessions.
- `activeChatId`: Currently selected assistant session ID.
- `backgroundMode`: Canvas background mode.
- `viewport`: Viewport transform; `x/y` is screen pan, `k` is zoom ratio.

## Node Structure

Each node is a `CanvasNodeData`:

```ts
type CanvasNodeData = {
  id: string;
  type: "image" | "text" | "config" | "video";
  title: string;
  position: { x: number; y: number };
  width: number;
  height: number;
  metadata?: CanvasNodeMetadata;
};
```

Common fields:

- `id`: Node ID.
- `type`: Node type; currently image, text, generation config, video.
- `title`: Node title.
- `position`: Canvas world coordinates, not screen coordinates.
- `width` / `height`: Node dimensions in canvas world coordinates.
- `metadata`: Node content and business state.

Currently used `metadata` fields:

```ts
type CanvasNodeMetadata = {
  content?: string;
  prompt?: string;
  status?: "idle" | "success" | "loading" | "error";
  errorDetails?: string;
  fontSize?: number;
  generationMode?: "text" | "image" | "video";
  model?: string;
  size?: string;
  count?: number;
  naturalWidth?: number;
  naturalHeight?: number;
  freeResize?: boolean;
  isBatchRoot?: boolean;
  batchRootId?: string;
  batchChildIds?: string[];
  primaryImageId?: string;
  imageBatchExpanded?: boolean;
  inputOrder?: string[];
  storageKey?: string;
  mimeType?: string;
  bytes?: number;
};
```

Usage by node type:

- Image node: `content` is the current displayable image URL, usually a `blob:` URL; `storageKey` points to the local image Blob; `naturalWidth/naturalHeight/bytes/mimeType` store original image info.
- Video node: `content` is the current playable video URL, usually a `blob:` URL; `storageKey` points to the local video Blob; `bytes/mimeType` store file info.
- Text node: `content` stores text content; `fontSize` stores font size; `prompt/status/errorDetails` store generation state.
- Generation config node: `generationMode/model/size/count/inputOrder` store generation config; `generationMode` can be text, image, or video; upstream inputs are computed via `connections`.
- Image group node: The root node uses `isBatchRoot/batchChildIds/primaryImageId/imageBatchExpanded` to record batch generation results; child nodes use `batchRootId` to reference the root.

## Connection Structure

Each connection is a `CanvasConnection`:

```ts
type CanvasConnection = {
  id: string;
  fromNodeId: string;
  toNodeId: string;
};
```

Connections only store node IDs, not port coordinates. The render path is computed from node positions and dimensions.

Deleting a node also deletes connections where it is the source or target. Deleting an image group root node also deletes all child nodes.

## Assistant Session Structure

Assistant sessions are stored within the canvas project:

```ts
type CanvasAssistantSession = {
  id: string;
  title: string;
  messages: CanvasAssistantMessage[];
  createdAt: string;
  updatedAt: string;
};
```

Message structure:

```ts
type CanvasAssistantMessage = {
  id: string;
  role: "user" | "assistant";
  mode: "ask" | "image";
  text: string;
  isLoading?: boolean;
  references?: CanvasAssistantReference[];
  images?: CanvasAssistantImage[];
};
```

Image references and assistant-generated images also follow the same image storage rules:

- The `dataUrl` field may currently be a `blob:` URL or a legacy `data:image/...`.
- When `storageKey` exists, the image Blob is read via `storageKey`.
- Before sending to the AI API, if the API requires base64, the Blob URL is temporarily converted to a data URL via `imageToDataUrl`.

## Image Write Flow

All new images should be written via `uploadImage(input)`:

1. Pass in a `Blob` or data URL.
2. Internally converted to `Blob`.
3. Generate `storageKey` in format `image:<id>`.
4. Write Blob to `image_files`.
5. Create `blob:` URL and cache in memory `objectUrls`.
6. Read image dimensions and return:

```ts
type UploadedImage = {
  url: string;
  storageKey: string;
  width: number;
  height: number;
  bytes: number;
  mimeType: string;
};
```

Image nodes write via `imageMetadata(image)`:

```ts
{
  content: image.url,
  storageKey: image.storageKey,
  status: "success",
  naturalWidth: image.width,
  naturalHeight: image.height,
  bytes: image.bytes,
  mimeType: image.mimeType
}
```

Therefore, `content` is only suitable for the current browser session display and should not be used as a long-term file identifier; the long-term identifier is `storageKey`.

## Image Reading and Legacy Data Migration

On canvas open, image hydration runs:

- If the image node has a `storageKey`, read the Blob via `resolveImageUrl(storageKey, fallback)` and generate a new `blob:` URL.
- If the image node has no `storageKey` but `content` is a legacy `data:image/...`, call `uploadImage(content)` to migrate to `image_files` and add `storageKey`.
- Reference images and generated images in assistant messages also run the same logic.

My Assets also runs migration on read:

- With `storageKey`: restore `coverUrl` and `data.dataUrl` displayable URLs.
- Without `storageKey` but has base64: write to `image_files`, then update `storageKey` in the asset.

## Image Removal and Cleanup

Images are not immediately deleted node-by-node when nodes are deleted. Instead, reference cleanup is performed:

1. Deleting nodes, clearing canvases, deleting canvases, deleting assets, or deleting assistant sessions triggers `cleanupImages`.
2. `cleanupImages` collects all `storageKey`s still referenced by canvas projects, assets, and additional data.
3. `cleanupUnusedImages` iterates all images in `image_files`.
4. Images not in the reference set are deleted.
5. On deletion, `URL.revokeObjectURL` is called and the entry is removed from the memory cache `objectUrls`.

This approach avoids accidentally deleting an image that is referenced by the canvas, assets, or assistant simultaneously.
