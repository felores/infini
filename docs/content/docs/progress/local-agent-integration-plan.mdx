# Local Agent Integration Plan

This document plans how individual users accessing the online canvas web page can connect Codex / Claude Code on their own computer and let the Agent operate the current canvas through conversation and tool calls.

## Goals

- After opening the online canvas, users can start a local service to connect to the current canvas.
- Users can operate the canvas via MCP in the Codex terminal.
- Users can also converse with local Codex / Claude Code in the web sidebar.
- The sidebar needs to display regular messages, streaming output, tool calls, tool results, and error prompts.
- The online service does not store user local Codex / Claude Code login state, API keys, or local file permissions.

## Core Conclusion

The browser page cannot directly start local processes and should not directly control local Codex / Claude Code. The recommendation is to add a `canvas-agent` service running on the user's machine:

```txt
Online Canvas Web <-> Local canvas-agent <-> Codex / Claude Code
                          |
                          +-> MCP Server
                          +-> Canvas Connection
```

`canvas-agent` is the local trust boundary, responsible for starting or connecting Codex / Claude Code, exposing MCP tools, maintaining canvas connections, and forwarding sidebar conversation events.

## Recommended Directory

When implementing, add a standalone npm package in the project root:

```txt
canvas-agent/
  package.json
  tsconfig.json
  src/
    index.ts
    config.ts
    http-server.ts
    canvas-session.ts
    mcp-server.ts
    agents.ts
    schemas.ts
    tools.ts
    types.ts
```

Reasons:

- This is a Node service running on the user's machine, not part of the online backend or the pure frontend page.
- It can be published as an npm package for users to start via `npx` or global install.
- Codex SDK, Codex MCP, and Claude SDK are all better suited for a local Node process.
- HTTP routes use Express, MCP protocol layer uses official `@modelcontextprotocol/sdk`, tool parameters use `zod`, avoiding hand-written protocols and loose JSON.

## Local Startup

The published `0.1.0` is only for plugin MCP; this repo and local Agent prioritize running current source:

```bash
bun --cwd canvas-agent src/index.ts
```

Output after startup:

```txt
Infinite Canvas Agent
Local URL: http://127.0.0.1:17371
Connect token: xxxxxx
```

The web sidebar connects to the local service after filling in or auto-discovering `http://127.0.0.1:17371` and token.

## Frontend Capabilities Needed

The frontend does not expose HTTP endpoints to the local service directly; the web page proactively connects to the local Canvas Agent. The current MVP uses SSE to receive local events and HTTP POST to report canvas state and tool results:

```txt
Web -> http://127.0.0.1:17371/events?token=xxx
Web -> http://127.0.0.1:17371/canvas/state?token=xxx
Web -> http://127.0.0.1:17371/canvas/result?token=xxx
```

The frontend needs a canvas Agent control layer that internally calls existing store / hook, without letting external systems directly operate React components:

```ts
canvasAgent.getState()
canvasAgent.getSelection()
canvasAgent.applyOps(ops)
canvasAgent.focusNodes(ids)
canvasAgent.exportSnapshot()
```

Recommended minimal operation set:

- `add_node`: Add image, text, audio, video, generation config nodes.
- `update_node`: Update node position, size, content, and config.
- `delete_node`: Delete nodes.
- `connect_nodes`: Connect two nodes.
- `set_viewport`: Pan or zoom the current viewport.
- `select_nodes`: Select nodes.

Tool parameters should use canvas business JSON, not simulated mouse clicks or screen-coordinate automation.

When MCP reads canvas state, it returns a summary by default, not stuffing full images, videos, audio, or very long base64 content to the Agent.

## MCP Tool Design

`canvas-agent` has a built-in MCP Server for Codex CLI to connect:

```txt
Codex CLI <-> canvas-agent MCP <-> Current Web Canvas
```

Recommended first batch of MCP tools:

- `canvas_get_state`: Read current canvas nodes, connections, selection, and viewport summary.
- `canvas_apply_ops`: Batch-execute canvas operations.
- `canvas_get_selection`: Read currently selected nodes.
- `canvas_export_snapshot`: Export current canvas snapshot for Agent layout understanding.
- `canvas_create_text_node`: Quick-create a text node.
- `canvas_create_image_prompt_flow`: Quick-create a prompt text node and image generation config node.

User local Codex config example:

```bash
codex mcp add infinite-canvas -- npx -y @basketikun/canvas-agent@0.1.0 mcp
```

For this repo debugging (use your machine's absolute path for actual config):

```bash
codex mcp add infinite-canvas -- node /path/to/infinite-canvas/canvas-agent/dist/index.js mcp
```

The web sidebar assistant uses Canvas Agent's conversation channel. Canvas Agent uses official `@openai/codex` CLI's `codex app-server --stdio` to start and reuse the same thread; on startup it injects the `infinite-canvas` MCP config and sets `default_tools_approval_mode` to `approve` to prevent Codex's own MCP approval from blocking the sidebar; actual canvas modifications are confirmed a second time by the web sidebar.

Sidebar image attachments are sent via HTTP to the local Canvas Agent, which temporarily writes them to local files and passes them to Codex as app-server `localImage` input; the MVP shows attachment size in the input area, with a single request body limit of 30MB.

## Sidebar Assistant Design

After the sidebar assistant connects to `canvas-agent`, Canvas Agent selects the Agent Adapter:

```txt
Sidebar -> canvas-agent -> Codex app-server stdio
Sidebar -> canvas-agent -> Claude Code CLI / Claude Agent SDK
```

The sidebar prioritizes displaying Codex app-server native structured events. The frontend only consumes necessary events:

```ts
type AgentEvent =
  | { type: "thread.started"; thread_id: string }
  | { type: "turn.started" }
  | { type: "item.started" | "item.updated" | "item.completed"; item: ThreadItem }
  | { type: "turn.completed"; usage: Usage }
  | { type: "turn.failed"; error: { message: string } };
```

Claude integration will have its own Adapter later; currently the frontend only shows Codex by default.

## Codex Integration Priority

Prioritize using official `@openai/codex` CLI's `codex app-server --stdio`.

- Codex app-server outputs `item/agentMessage/delta`; Canvas Agent converts to `item.updated`, and the frontend uses the same message for real streaming rendering.
- Image attachments use app-server `localImage` input, no hand-written multimodal protocol.

References:

- https://github.com/openai/codex/tree/main/codex-rs/app-server
- https://developers.openai.com/codex/codex-manual.md

## Claude Code Integration Priority

Claude Code MVP first calls local Claude Code CLI streaming JSON output, with `canvas-agent` uniformly transforming events and tool calls; later upgrade to Claude Agent SDK.

If you want Claude Code to also operate the current canvas, configure the same MCP for Claude Code:

```bash
claude mcp add --scope user --transport stdio infinite-canvas -- npx -y @basketikun/canvas-agent@0.1.0 mcp
```

Canvas Agent allows `mcp__infinite-canvas__*` by default when calling Claude Code, to avoid print mode being blocked by tool approval; actual canvas modifications are still confirmed by the web sidebar.

References:

- https://docs.anthropic.com/en/docs/claude-code/sdk
- https://code.claude.com/docs/en/agent-sdk/typescript

## Security Boundaries

- Canvas Agent only listens on `127.0.0.1` by default.
- Canvas Agent generates a token on startup; web connections must carry the token.
- Canvas Agent only accepts allowed Origins, defaulting to the user's currently open canvas domain.
- Canvas operations default to showing tool calls in the sidebar first; `canvas_apply_ops` requires user confirmation before execution, and retains an undo entry for the last tool operation.
- Codex / Claude login state, API keys, and user local file paths are not uploaded to the online service.
- The online web page only stores the Canvas Agent address and necessary preferences, not local keys.

## Implementation Phases

### Phase 1: Codex Terminal Operates Canvas

1. Add the `canvas-agent/` npm package.
2. Canvas Agent provides SSE/HTTP connection; the web page connects and registers the current canvas.
3. Frontend adds the `canvasAgent` control layer.
4. Canvas Agent has a built-in MCP Server.
5. Codex calls `canvas_get_state` and `canvas_apply_ops` via MCP.

### Phase 2: Web Sidebar Connects Codex

1. Frontend adds the local Agent sidebar.
2. Canvas Agent integrates Codex via official `@openai/codex` CLI's `codex app-server --stdio`, reuses the same thread, injects `infinite-canvas` MCP, and displays structured event stream.
3. The run log retains only key Codex events, avoiding flooding with streaming intermediate updates.
4. The sidebar displays message flow, tool calls, tool results, errors, and image attachment size hints.

### Phase 3: Claude Code Integration

1. Canvas Agent adds Claude Code CLI Adapter.
2. Unify Claude Code message and tool call events.
3. Frontend sidebar adds Agent type selection.

### Phase 4: Experience Polish

1. Add local connection status, reconnection, and token updates.
2. Optimize tool call confirmation, undo, and batch tool queue experience.
3. Add common canvas action templates.
4. Add npm publishing and user installation docs.

## MVP Acceptance Criteria

- After running `bun --cwd canvas-agent src/index.ts`, the online canvas shows connected.
- Users can create text nodes, move nodes, and connect nodes in the Codex CLI.
- The web sidebar can send a message to local Codex and display streaming replies.
- The web sidebar only shows local Codex by default, with structured replies.
- The sidebar can display one `canvas_apply_ops` tool call and result.
- After disconnecting Canvas Agent, the web shows a clear local connection failure message.
