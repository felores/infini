# Local Codex Canvas Connection Principles

The core of connecting local Codex to the canvas is not having Codex directly control the browser, nor simulating mouse clicks or DOM manipulation, but running a `Canvas Agent` as a bridge service on the user's computer.

```txt
Codex / Browser Sidebar
        |
        v
Local Canvas Agent
        |
        v
Browser Canvas React State
```

The actual canvas data remains in the browser; nodes, connections, selection, and viewport are all maintained by the frontend React state. Codex can only send "read canvas" or "execute canvas operations" requests through the tool protocol exposed by `Canvas Agent`.

## Two Usage Directions

### 1. Local Codex Actively Connects to Canvas

The entry point for this direction is in Codex.

```txt
Local Codex
  -> infinite-canvas MCP tools
  -> Local Canvas Agent
  -> Browser Canvas
```

The user initiates a request in local Codex, such as reading the canvas, creating text nodes, connecting nodes, or triggering generation. Codex calls the `infinite-canvas` MCP tools, which then request the local `Canvas Agent`'s `/api/tools` endpoint.

Common tools include:

- `canvas_get_state`: Read the current canvas summary.
- `canvas_get_selection`: Read the current selection.
- `canvas_apply_ops`: Batch-execute canvas operations.
- `canvas_create_text_node`: Create a text node.
- `canvas_generate_image` / `canvas_generate_text`: Create a generation flow and trigger generation.
- `canvas_update_node` / `canvas_connect_nodes`: Update or connect nodes.

This path is suitable for directly having Codex operate on the currently open canvas in a Codex conversation.

### 2. Browser Right Panel Actively Opens Codex

The entry point for this direction is in the browser.

```txt
Browser Right Codex Panel
  -> Local Canvas Agent
  -> Codex app-server --stdio
  -> infinite-canvas MCP tools
  -> Local Canvas Agent
  -> Browser Canvas
```

After the user enters a prompt in the web right panel, the browser sends the request to the `Canvas Agent`'s `/agent/codex/turn` endpoint. `Canvas Agent` starts or reuses `codex app-server --stdio` locally and injects the same set of `infinite-canvas` MCP tools into this Codex session.

So even though the entry point is the browser right panel, when actually modifying the canvas, it still goes through the same tool chain:

```txt
Codex -> MCP -> Canvas Agent -> Browser Canvas
```

The right panel simply brings "open a Codex session, send a prompt, display streaming events" into the web page.

## How the Browser and Canvas Agent Connect

`Canvas Agent` listens on the local address by default:

```txt
http://127.0.0.1:17371
```

On startup it generates a `Connect token` that the web page must carry when connecting. The browser establishes an SSE long connection:

```txt
GET /events?token=xxx&clientId=xxx
```

On successful connection, `Canvas Agent` sends a `hello` event and the browser updates its status to connected.

## How Canvas State Syncs to Codex

The browser proactively syncs the current canvas snapshot to `Canvas Agent`:

```txt
POST /canvas/state?token=xxx&clientId=xxx
```

The snapshot contains:

- Current canvas ID and title
- Node list
- Connection list
- Currently selected node IDs
- Current viewport position and zoom

`Canvas Agent` keeps the latest snapshot in memory. When Codex calls `canvas_get_state`, it reads this latest snapshot synced from the browser, not the browser DOM directly.

## How Codex Operates the Canvas

Taking creating a text node as an example, the flow is:

```txt
Codex calls canvas_create_text_node
  -> MCP POST /api/tools
  -> Canvas Agent converts to canvas_apply_ops
  -> Canvas Agent sends tool_call via SSE
  -> Browser executes add_node
  -> Browser POST /canvas/result
  -> Canvas Agent returns result to Codex
```

`Canvas Agent` converts some high-level tools into unified canvas operations:

```json
{
  "ops": [
    {
      "type": "add_node",
      "nodeType": "text",
      "metadata": {
        "content": "Text content",
        "status": "success"
      }
    }
  ]
}
```

After receiving the `tool_call`, the browser calls the frontend's `applyCanvasAgentOps` to execute `add_node`, `update_node`, `delete_node`, `connect_nodes`, `set_viewport`, `select_nodes`, etc. in React state. After execution, the result is sent back to `Canvas Agent` via `/canvas/result`.

## Tool Confirmation and Undo

To prevent Codex from directly modifying the canvas, the browser has a second-confirmation mechanism for `canvas_apply_ops` by default. After receiving a write operation, the right panel first displays the pending tool call; the user confirms before it is actually applied to the canvas.

After execution, the frontend saves a snapshot of the canvas before the last operation for undoing the last Agent tool operation.

If tool confirmation is disabled, local Codex can execute canvas tool calls directly, but still through the same `Canvas Agent -> SSE -> Browser execution` chain.

## Security Boundaries

- `Canvas Agent` only listens on `127.0.0.1` by default.
- Web connections must carry the locally generated `Connect token`.
- After the first connection with the correct token, `Canvas Agent` records the web page Origin to prevent other sources from reusing the current local Agent.
- Codex does not directly access the browser DOM or hold internal canvas state.
- The online web page does not store local Codex login state, API keys, or local file permissions.
- Canvas write operations are ultimately executed by the browser frontend, so confirmation, undo, and permission control can be done on the frontend.

## Summary

The essence of local Codex connecting to the canvas is:

```txt
Codex calls MCP tools, Canvas Agent bridges locally, and the browser receives tool calls via SSE and actually modifies the canvas in React state.
```

Reading state:

```txt
Browser snapshot -> Canvas Agent memory -> MCP tools -> Codex
```

Writing operations:

```txt
Codex -> MCP tools -> Canvas Agent -> SSE tool_call -> Browser applyOps -> /canvas/result -> Codex
```
