How Infini connects, and where KIE and Sinapso fit

A code-grounded tour of the browser application, provider calls, local HTTP agent, MCP bridge, CLI package, persistence, external services, and the shortest safe path to KIE and cross-repository orchestration.

Codebase Memory ready Infini graph: 2,731 nodes / 5,348 edges Sinapso graph: 2,702 nodes / 6,618 edges KIE graph: 735 nodes / 1,147 edges Commit bc70acc6 Working-tree snapshot: 130 tracked changed files

The short answer

Infini already has two useful seams. The channel system handles OpenAI-compatible and native Gemini inference. The Canvas Agent exposes browser-owned canvas operations through token-gated loopback HTTP and stdio MCP.

Do not build a second KIE integration inside Infini. The local kie-cli-mcp repository already owns KIE authentication, model schemas, task persistence, polling, CLI, stdio MCP, and remote Streamable HTTP MCP. For agent workflows, register KIE, Infini, and Sinapso as sibling MCP servers. Keep each product behind its existing boundary.
Minimal target

Three sibling MCPs, zero bridge code

Sinapso supplies knowledge, KIE generates media, and Infini receives the resulting URL as an asset or canvas node. Add a product adapter only if Infini’s own Generate button must call KIE without an agent host.

Call the external endpoint directly

The browser sends the configured bearer key to the selected provider. Text streams through SSE. Image usually returns inline data or URLs. Video polls until completion. This is Infini’s current browser-held-key trust model; each provider’s client-side key policy must still be checked. KIE explicitly forbids it.

KIE: reuse the system you already have

kie-cli-mcp is already the modular KIE boundary. Its private core owns schemas, API payloads, SQLite task history, and polling; public CLI and MCP packages expose the same registry.

Shared core
One tool registry

CLI and MCP derive commands, schemas, model routing, and errors from TOOL_REGISTRY.

Public surfaces
CLI + MCP

kie-cli, stdio MCP, and optionally bearer-authenticated Streamable HTTP MCP are implemented and published.

Task lifecycle
SQLite + wait tool

wait_for_task polls or uses callback rendezvous and returns final result URLs.

Recommended agent workflow: no repository changes

Agent host (Codex / Claude / OpenCode) 1. Sinapso MCP → search notes and passages 2. KIE MCP → nano_banana_image / bytedance_seedance_video 3. KIE MCP → wait_for_task → result_urls[] 4. Infini MCP → assets_add or canvas_create_node(result URL) KIE_AI_ENABLED_TOOLS=image-or-video-tool,wait_for_task KIE_AI_API_KEY stays in the KIE MCP process

This is the shortest path because each repository already exposes the needed MCP surface. Infini does not need to understand KIE task states, and KIE does not need to understand canvas state. Tool filtering keeps the KIE schema footprint small.

Using Infini from Sinapso

The local repository is named sinapso. It should remain the knowledge surface while the existing KIE MCP owns generation and Infini MCP owns canvas operations.

1 · Recommended

Three sibling MCP servers

Register Sinapso, KIE, and Infini in Codex, Claude, OpenCode, or another MCP host. The host moves only selected context and result URLs between them.

  • No code changes
  • Lowest coupling
  • Secrets stay with owning process
2 · Scriptable

CLI pipeline

Use sinapso call for knowledge, kie-cli --json plus wait_for_task for media, then Infini’s token-gated /api/tools endpoint.

  • Good for deterministic scripts
  • No MCP client required
  • Requires local process and token handling
3 · Only if product UI needs it

One explicit bridge tool

Add a small, allowlisted Sinapso registry entry that invokes one Infini operation, or the reverse. Do this only for a concrete user workflow that cannot rely on an external MCP host.

  • Tighter coupling
  • Needs lifecycle and token config
  • Avoid generic MCP passthrough

Example cross-repository sequence

1

Agent calls Sinapso search_notes, search_passages, or a registry-backed read tool.

2

Agent synthesizes only the selected notes or passages into prompts, references, or a visual plan.

3

Agent calls the selected KIE generation tool, then wait_for_task.

4

Agent receives result_urls without implementing KIE polling itself.

5

Agent calls Infini assets_add or canvas_create_node; the connected browser stores or renders the generated media.
