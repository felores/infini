Core model

OMP is steerable, but routing and model selection are different systems.

Recreate your OpenCode governance by combining reusable model roles, named task-agent definitions, and a narrow routing policy appended to OMP's default system prompt.

Use APPEND_SYSTEM.md for routing policy. It keeps OMP's built-in tool, skill, workflow, and project-context guidance. SYSTEM.md replaces that stable block and is only appropriate when you intentionally own the complete base prompt.

Agent type is explicit

OMP's task call contains one exact agent name. It is not a hidden classifier and does not route from a model role automatically.

Model roles are reusable profiles

smol, task, slow, designer, advisor, and custom tags select a model plus optional default effort.

Tool permissions are the boundary

A prompt guides behavior. An agent's allowed tools make boundaries real, such as a verifier that can run checks but cannot edit code.

Reasoning

    Batch boundary: one OMP task call has one agent type for every item in its tasks[] batch. A workflow needing extraction, review, and verification requires separate calls. The task item's role is a prompt identity and UI label, not an agent selector.

    Minimal rollout order
    Start with a few high-signal specialists. Add agents only when their boundaries are genuinely distinct.
    
        
    
          
    Set model roles
    Configure smol, task, slow, designer, plan, and advisor in /models. Assign effort based on the job, not a generic price tier.
    
          
    Add only missing specialists
    Use bundled explore, task, reviewer, designer, and plan first. Add custom verifier, writer, or extractor only where their tool boundary adds value.
    
          
    Append the routing policy
    Write concise scenario rules in APPEND_SYSTEM.md. Do not turn it into a long rigid workflow that blocks simple, direct work.
    
          
    Measure signal
    Keep an agent only when it improves a real decision, catches defects, or saves meaningful context. Remove noisy profiles and disable advisor if its every-turn review does not earn its cost.
    
        
    
      
    

      
    
        Important limits
        
    Prompts steer. Definitions and tools enforce.
    
        
    The reliable OMP system is not a giant supervisor prompt. It is small policy text backed by explicit agent definitions, constrained tools, and focused task assignments.
    
        
    
          
    Cannot A model role choose an agent
    modelRoles.task determines a model profile for generic task work. It cannot mean “run verifier when tests are needed.” The primary agent must explicitly select agent: "verifier".
    
          
    Cannot A system prompt reliably forbid a tool
    Use the agent definition's tool list. A read-only reviewer should not have edit or write; a verifier may have bash for focused checks but no editing tools.
    
          
    Cannot Mix agent types inside one task batch
    A batch has one agent field. Split workflows into separate calls when the work crosses capability boundaries.
    
          
    Careful Overriding a bundled agent replaces it
    Project agent discovery wins over user, plugin, and bundled definitions by exact name. If you define reviewer, you replace its entire bundled prompt and tool policy. Prefer a new name such as deep-reviewer unless you intend to maintain that override.
    
          
    Use Agent profiles as capability contracts
    Write assignments that name the required output, constraints, files, non-goals, and verification. The profile decides model, effort, and tool boundary. The assignment decides the actual work.
